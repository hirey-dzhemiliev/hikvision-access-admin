# Журнал событий

## Назначение

Журнал нужен для того, чтобы видеть:

- кто проходил;
- когда это произошло;
- через какую панель;
- каким способом выполнен доступ;
- есть ли снимок события.

Это именно журнал проходов, а не общий технический лог устройства.

## Какие события нужны

В основном журнале нужны только эксплуатационные события доступа:

- успешный проход;
- отказ в доступе;
- удалённое открытие;
- проход по карте;
- проход по лицу;
- проход по QR, если используется.

Технические и диагностические события в основной журнал не выводятся.

## Что хранится по событию

- время события;
- панель;
- `employee_id`, если он известен;
- имя сотрудника;
- номер комнаты;
- номер карты;
- способ прохода;
- результат;
- путь к снимку;
- raw JSON события.

## Сопоставление с локальной базой

Сначала сопоставление идёт по `employee_id`, затем по `card_number`.

Если сотрудник не найден, событие всё равно сохраняется, но помечается как несопоставленное.

## Что должно быть удобно в UI

- список последних проходов;
- фильтр по дате;
- фильтр по сотруднику;
- фильтр по панели;
- фильтр по результату;
- фильтр по способу;
- миниатюра фото;
- переход в карточку сотрудника;
- переход из сотрудника в историю его проходов.

## Источник данных

Приложение получает события через:

- `GET /ISAPI/Event/notification/alertStream`

События принимаются, нормализуются и сохраняются в основной `SQLite`-базе приложения.
