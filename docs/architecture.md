# Архитектура и хранение данных

## Архитектурный принцип

Приложение разделено на несколько слоёв:

- web-layer;
- слой хранения данных;
- слой синхронизации и фоновых worker'ов;
- отдельный клиент Hikvision для работы с ISAPI.

Это позволяет:

- не смешивать UI и сетевую интеграцию;
- не смешивать маршруты, `SQLite` и background workers в одном месте;
- безопаснее добавлять новые операции;
- переиспользовать логику для нескольких панелей.

## Хранение данных

Принята следующая схема:

- `SQLite` хранит сотрудников, панели, журнал, sync-историю и кэш;
- фотографии и снимки лежат на диске;
- системные настройки и пользователи админки хранятся в одном `YAML`-файле.

## Что хранится в SQLite

- сотрудники;
- панели;
- журнал проходов;
- кэш `sync preview`;
- история синхронизаций;
- действия синхронизации.

## Что хранится на диске

- фото сотрудников;
- снимки событий прохода.

## Что хранится в YAML

- пользователи админки;
- секрет для сессий;
- системные параметры запуска;
- пути к БД и медиа;
- параметры журнала и ротации.

## SQLite runtime-настройки

При старте приложения используются:

- `journal_mode = WAL`
- `synchronous = NORMAL`
- `busy_timeout = 5000`
- `foreign_keys = ON`
- `temp_store = MEMORY`
- `wal_autocheckpoint = 1000`
